1. System Architecture `☁️ Server`

1.1 Federated Private Clusters `☁️ Server`

Kiến trúc Cluster thay thế VPS đơn lẻ (Single Point of Failure):

  • Encryption: Toàn bộ Payload mã hóa bằng Company_Key trước khi rời thiết bị.
  • Storage: Cluster gồm 3–5 Nodes chạy Erasure Coding (Sharding dữ liệu). Nếu 1 Node sập, dữ liệu tự phục hồi từ các mảnh còn lại.
  • Routing: Client kết nối trực tiếp tới IP của Company Cluster, không đi qua Cloud công cộng.

Cluster Scope — Hạ tầng 4-trong-1:

Thành phần Chức năng
MLS Backbone Phân phối khóa & định tuyến tin nhắn cho nhóm 5000+ người. Lưu trữ Encrypted Log Streams (Zero-Knowledge).
HA TURN Cluster Floating IP, tự động Failover trong 3s. Chịu tải hàng nghìn cuộc gọi HD/Video Conference.
Execution Environment Thực thi backend cho Mini-Apps nội bộ, dữ liệu P2P chỉ nằm trên RAM.
Interop Hub Gateway mã hóa dữ liệu từ SAP/Jira/CRM chuẩn E2EE trước khi đẩy xuống Client.

1.2 Tổng thể Kiến trúc: "Local-First Processing & Blind Routing" `💻 Laptop` `🖥️ Desktop` `☁️ Server`

Nguyên tắc cốt lõi: Server (VPS/Cluster) chỉ luân chuyển gói Byte mã hóa. Mọi tác vụ nặng (Tìm kiếm, Lưu trữ, Sinh khóa) đều bị đẩy về Client (Desktop App).

Trụ cột 1: Zero-Ticket Identity Recovery

  • Giải pháp: Multi-Device Cross-Signing (Mặc định) + Mailbox Re-binding (Dự phòng).
  • Cơ chế: Khi Laptop hỏng, Mobile ký xác thực (Cross-Sign) cho thiết bị mới. Nếu mất cả 2, IT Admin revoke và cấp Invite Token mới. Nhân viên chấp nhận mất tin nhắn > 14 ngày.

Trụ cột 2: Zero-Knowledge Search

  • Cơ chế (Desktop): Background Worker lập chỉ mục toàn bộ tin nhắn đã giải mã bằng SQLite FTS5 (millisecond search). Bản báo cáo phức tạp chạy qua Local Edge RAG (Embedding WASM/Rust).
  • Bảo đảm: Không metadata nào rò rỉ lên mạng.
🔺 IMPORTANT

Lead Dev Fix #3 — Giới hạn FTS5 trên Mobile (Apple DAS / Android Doze): Apple DAS (Duet Activity Sync) và Android Doze Mode không bao giờ cho phép tiến trình nền chạy FTS5 Indexing cày nát CPU và Pin. Nếu cố ép, app sẽ bị đưa vào Blacklist tiết kiệm pin của OS. Hai nền tảng phải có chiến lược lập chỉ mục tìm kiếm tách biệt.

1.4 Tiered Storage Architecture — Kiến trúc Lưu trữ Phân tầng `💻 Laptop` `🖥️ Desktop` `☁️ Server`

Bài toán: Kiến trúc Local-First yêu cầu dữ liệu nằm trên Client để hoạt động offline. Nhưng nếu Client tải toàn bộ 50GB lịch sử tin nhắn, video, file log thì app sẽ ngốn hết RAM, SSD máy trạm "báo đỏ" chỉ sau 1 tháng, và khởi động app bị nghẽn. Giải pháp: Phân tầng lưu trữ (Tiered Storage) kết hợp Lazy-Loading — máy trạm chỉ lưu phần nóng nhất, phần lạnh nằm nguyên trên Cluster.

A. Lean Edge Caching — Zero-Byte Stubs + Selective Offline

Triết lý Lean: Offline-First không có nghĩa là ôm đồm dữ liệu. 99% thời gian thiết bị có mạng — tự động tải hàng GB file vào máy nhân viên là lãng phí tài nguyên phần cứng. Hệ thống chỉ tải khi có lý do cụ thể.

Ba tầng quyết định tải file:

Tầng Cơ chế Ai kiểm soát Khi nào tải
Tầng 1 — Auto (Pinned) Rust Core tự động kéo full ciphertext về local Admin / Manager File được đánh dấu Pin trong Channel
Tầng 2 — On-demand User click "Tải xuống" / "Xem trước" hoặc mở TeraVault User Khi cần, ngay lập tức via Cluster hoặc LAN P2P
Tầng 3 — Make Available Offline Background worker kéo toàn bộ folder về local User tự chọn Trước khi lên máy bay / ra giàn khoan / vào khu vực mất mạng

Thực thi on-demand (Tầng 2 — User click):

C. Convergent Encryption — Salted MLE + Contextual Heuristics Engine

Vấn đề của MLE cổ điển: Key = SHA-256(file_content) dễ bị Known-Plaintext Attack (KPA) — hacker tự hash file phổ biến (mẫu đơn xin nghỉ phép chuẩn, file cài phần mềm công khai) → suy ra key → giải mã ciphertext. TechSpec cũ thừa nhận "MLE chỉ phù hợp cho file không nhạy cảm" nhưng không có cơ chế tự động phân loại.

Giải pháp nâng cấp: Kết hợp Contextual Heuristics Engine (bộ định tuyến tĩnh, zero-latency) + Salted MLE (HMAC-SHA256 keyed bởi Channel_Key) → đạt đồng thời 3 mục tiêu: tiết kiệm 80–90% dung lượng, chống KPA tuyệt đối từ bên ngoài, và zero-friction với người dùng.

Nguyên tắc loại trừ: Mặc định mọi file dùng StrictE2EE — trừ khi thỏa mãn điều kiện "Được phép Salted MLE".

C2. Tầng Cryptography — Salted MLE (Chống KPA + Epoch-Independent Dedup)
🔺 IMPORTANT

Lead Dev Fix #2 — Xung đột giữa Salted MLE và MLS Ratchet: Nếu dùng Channel_Key (xoay theo MLS Epoch) làm salt để tính CAS_Hash, sau mỗi Epoch Rotation (thêm/bớt thành viên), cùng một file sẽ tạo ra CAS_Hash khác hoàn toàn so với lần gửi trước. Deduplication xuyên thời gian thất bại 100%. Giải pháp: Tách biệt Static_Dedup_Key (bất biến, chỉ dùng để tính hash) với Channel_Key (xoay theo Epoch, dùng để mã hóa nội dung).

       ↓

       ↓

4.4 C. Cấu trúc Mesh Bất Đối Xứng (Asymmetric Topology) `💻 Laptop` `🖥️ Desktop` `📱 Android`

Cấp độ Thiết bị Vai trò Kết nối Network

Thăng cấp kết nối (Connection Upgrading):

BLE 5.0 thực tế chỉ đạt ~1-2 Mbps, không gánh nổi các file/ảnh nặng 5MB. Việc gửi luồng file quá lớn sẽ gây ngập lụt toàn bộ băng thông Mesh. Phải kết hợp quy luật Thăng cấp (Upgrading):

  2. Tin nhắn Text (< 4KB): Truyền thẳng qua sóng BLE.
  3. File nặng (> 4KB): Bắt hệ điều hành mở kênh Wi-Fi Direct (cho Android) hoặc AWDL P2P (cho iOS), lập đường truyền ~250Mbps trong nền. Truyền xong file sẽ đóng băng luồng lại ngay lập tức để tiết kiệm Pin.

5.4.C Payload Puncturing — Khả năng phục hồi luồng file `⚙️ Core` `💻 Laptop` `🖥️ Desktop` `📱 Mobile`

🔺 IMPORTANT

Dev Fix #3 — Cảm hứng Wi-Fi 7 Puncturing: Wi-Fi 7 bỏ qua các phần phổ tần bị nhiễu (ví dụ: lò vi sóng đang chạy làm nhiễu 2.4GHz) thay vì hạ toàn bộ băng thông. Giải pháp: PuncturingStream — đánh dấu chunk bị nhiễu là "Punctured", tiếp tục bơm chunk tiếp mà không ngắt kết nối; vá chunk bị thủng ở cuối phiên.

Stream file 2GB = 1024 chunk × 2MB

  Chunk  1 → ✅ OK
  Chunk  2 → ✅ OK
  ...

               → KHÔNG hủy kết nối
               → Đánh dấu "Punctured", tiếp tục
  Chunk 46 → ✅ OK
  Chunk 47 → ✅ OK
  ...
  Chunk 1024 → ✅ OK

  Sau khi stream xong:
    Bước phục hồi: Chunk 45 được gửi lại qua BLE Lane 1
    hoặc chờ nhiễu qua → Wi-Fi Direct "vá" lại cuối phiên
    → File toàn vẹn 100%, không cần resend từ đầu

8.5 Cứu hộ Database & Chống Corrupt (Suspend-Aware SQLite) `📱 Mobile`

Tử huyệt Mobile I/O: Khác với Desktop, OS trên Mobile (iOS Jetsam / Android Doze) có thể "bóp cổ" tiến trình (OOM-Kill) ngay lập tức khi App ở Background mà không có grace period. Nếu Lõi Rust đang thực thi Transaction ghi file .tapp, Database sẽ bị hỏng toàn bộ (Corrupted).

Để đảm bảo App luôn chết trong trạng thái an toàn (Crash-Safe), kiến trúc I/O trên Mobile bắt buộc tuân thủ 3 trụ cột:

Trụ cột 1: Ép buộc chế độ WAL (Write-Ahead Logging) & Checkpointing

Tuyệt đối không dùng cơ chế Rollback Journal mặc định của SQLite. Chế độ WAL ghi các thay đổi vào một file append-only (-wal), giữ nguyên tệp DB gốc trong lúc đang Transaction.

Trụ cột 2: OS Lifecycle FFI Bridge (Cơ chế Panic & Flush)

Rust Core không thể tự biết khi nào App bị đưa xuống nền. Tầng Native UI phải chịu trách nhiệm báo động cho Lõi Rust.

  • iOS (AppDelegate.swift): Bắt event applicationDidEnterBackground và applicationWillTerminate. Yêu cầu thêm Background Execution Time (3-5 giây) qua beginBackgroundTaskWithExpirationHandler.
  • Android (Application.kt): Bắt event onTrimMemory(TRIM_MEMORY_UI_HIDDEN).

Kiến trúc truyền tín hiệu:

Native UI gọi thẳng xuống FFI terachat_core_flush_io(). Hàm này kích hoạt wal_checkpoint để ép toàn bộ dữ liệu từ -wal chui vào file DB chính một cách nguyên tử (atomic).

Trụ cột 3: Chống phân mảnh Transaction (Micro-Batching)

Nguyên tắc lập trình DB trên Mobile: Không bao giờ mở một Transaction quá 50ms.

Nếu nhận 10,000 tin nhắn từ Server sau khi mất mạng, Lõi Rust phải chia nhỏ thành các "Micro-Batch" (Ví dụ: 200 tin nhắn/transaction). Nếu OS kill app giữa chừng, dữ liệu hỏng chỉ cấu thành ở batch đang xử lý dở dang, các batch trước đó vẫn được bảo lưu trọn vẹn.

12.1 Client-Side Technology & Runtime Environment

Rust Core (Tauri / FFI)

📱 Mobile · 💻 Laptop · 🖥 Desktop
  • Priority: Core
  • Cross-platform: Có
  • Lý do phù hợp: Biên dịch native thành thư viện tĩnh (librustcore.a cho iOS, .so cho Android) hoặc DLL/dylib cho Desktop. Quản lý toàn bộ State, Cryptography và Networking chung một lõi. Nhờ cấp phát bộ nhớ an toàn (Zero-cost abstractions), tốc độ giải mã E2EE đạt < 5ms, không gây "nghẽn" Main UI Thread của React Native hay Flutter.
  • Hạn chế: Trên iOS không hỗ trợ JIT dynamic execution khiến một số tính năng WASM phải fallback; trên Android yêu cầu quản lý luồng qua JNI boundary khắt khe để tránh thất thoát bộ nhớ (Memory leak) do Collector của Java/Kotlin không với tới. Laptop phải bắt sự kiện Lid-close để tự ép flush ghi đĩa trước khi Sleep.

Embedded Database (SQLCipher)

📱 Mobile · 💻 Laptop · 🖥 Desktop
  • Priority: Core
  • Cross-platform: Có
  • Lý do phù hợp: Database nhúng tại Client mã hóa bằng AES-256 transparent. Thiết lập bắt buộc chạy chế độ WAL (Write-Ahead Logging) cho toàn bộ .tapp local DB để bảo đảm vẹn toàn tệp. Mọi text, metadata, danh bạ đều được lưu tại File system local. Tận dụng SQLite FTS5 engine để truy vấn Full-Text Search.
  • Hạn chế: Trên Mobile, OS có quyền Jetsam/OOM Kill app bất cứ lúc nào khi ở background. Nếu bị Kill khi đang Write đĩa, Database sẽ Corrupt (xem giải pháp chi tiết tại §8.5 Cứu hộ Database). Rust Core bắt buộc phải lắng nghe tín hiệu applicationWillTerminate hoặc onTrimMemory để xả (flush) I/O tức thì trước khi cắt điện. Chế độ FTS5 Search cũng bị vô hiệu hóa trên điện thoại do OS hạn chế ngầm. Storage Window an toàn mức 7 ngày.

WASM Sandbox Runtime (Wasmtime)

💻 Laptop · 🖥 Desktop
  • Priority: Recommended
  • Cross-platform: Có
  • Lý do phù hợp: Môi trường thực thi bảo mật cho ứng dụng tiện ích nội bộ (.tapp) và OPA Security Policies. Bằng cơ chế Sandbox chuẩn, WASM Core cắt đứt toàn bộ tương tác file system và network trực tiếp. Giảm tải cho máy chủ bằng cách cho Client tự tính toán Logic trước khi đồng bộ (Local Edge Compute).
  • Hạn chế: Bị cấm hoàn toàn trên iOS do App Store Guidelines 2.5.2 (không chạy mã Dynamic JIT download từ server). Do đó, luồng UI động phải bị phân rã sang cấu trúc JSON thuần tĩnh (Adaptive Cards) để App Mobile tự render tĩnh, giảm độ linh hoạt.

Desktop Super Node (TeraShare 24/7 Local Seeder)

🖥 Desktop
  • Priority: Optional
  • Cross-platform: Có
  • Lý do phù hợp: Hệ thống kết nối nội bộ P2P tận dụng các máy Desktop (cắm nguồn 24/7) làm máy chủ trung chuyển (Super Node). Hỗ trợ tải file tốc độ Gbps trong LAN mà không cần vòng qua Server ngoài Internet.
  • Hạn chế: Trực tiếp chống chỉ định với Mobile (bị OS kill app) và Laptop (vì sẽ tự động ngắt kết nối khi gập màn hình / Sleep / Hibernate). Không thể ứng dụng trên Server thuần do thiếu cơ chế NAT Traversal trực diện vào mạng WLAN văn phòng.

Battery-Aware Processing & Wake-lock Throttling

📱 Mobile · 💻 Laptop
  • Priority: Core
  • Cross-platform: Có
  • Lý do phù hợp: Thiết kế cốt lõi dành riêng cho thiết bị phụ thuộc Pin hữu hạn (Laptop/Mobile). Tính năng tự động Tạm dừng FTS5 Indexing, hạ Bitrate WebRTC từ 720p xuống 360p, và giãn chu kỳ ping BLE khi OS báo mức Pin dưới 20%.
  • Hạn chế: Các OS (Windows/Linux) quản lý Power State bằng các API cực kỳ phân mảnh (như Lid-close event hay ACPI poll), yêu cầu Native Interop phức tạp để gọi đúng lúc. Chạy Desktop cắm sạc không cần kích hoạt kỹ thuật này.

Hardware Bound Encryption (Secure Enclave / TPM 2.0 / StrongBox)

📱 Mobile · 💻 Laptop · 🖥 Desktop
  • Priority: Core
  • Cross-platform: Có
  • Lý do phù hợp: Chống lại các rủi ro nhân bản Private Key thông qua việc sinh và niêm phong cặp khóa mật mã sâu bên trong chip phần cứng. Khi cần ký số (Authentication / Code Signing), CPU gửi instruction cho Enclave tính toán và chỉ trả về Chữ ký, nội dung Private Key không bao giờ lọt ra RAM của OS. Giúp chống Disk Cloning trên Desktop và Root/Jailbreak dump trên Mobile.
  • Hạn chế: API cấp phát khác biệt hoàn toàn (NCryptSignHash trên Win, SecKey trên macOS, NDK KeyStore trên Android). Rất khó maintain mã nguồn. Các chip này chủ yếu chỉ hỗ trợ thuât toán ECC P-256 phổ thông thay vì Curve25519 hay Ed25519 tiên tiến.

12.2 Middleware & Security Layer

OPA Policy Engine (Rego)

💻 Laptop · 🖥 Desktop · ☁ VPS Cluster · 🗄 Server Vật Lý
  • Priority: Core
  • Cross-platform: Có
  • Lý do phù hợp: Đánh giá cấp quyền phân tán ABAC (Attribute-Based Access Control). Cùng một tệp Policy (Quy tắc) được deploy cho Gateways ở Server (viết bằng Go) và biên dịch sang WASM đẩy xuống Desktop/Laptop. Hệ thống Client có thể tự Block hành vi lệnh Paste (DLP) ngay tại UI thread cục bộ với thời gian quyết định < 1ms thay vì phải gửi request hỏi Server.
  • Hạn chế: Cấu trúc Bundle Rego thường phình to. Phải nạp vào RAM trước khi Execute, tác động lớn đến memory footprint trên điện thoại di động do bị OS giới hạn (< 200MB background), do đó không triển khai OPA rule phức tạp trên Mobile Apps.

Message Layer Security (MLS) Protocol

📱 Mobile · 💻 Laptop · 🖥 Desktop · ☁ VPS Cluster · 🗄 Server Vật Lý
  • Priority: Core
  • Cross-platform: Có
  • Lý do phù hợp: Giao thức IETF chuẩn hóa E2EE cho nhắn tin đa nhóm. Cơ chế TreeKEM giải quyết bài toán mã hóa O(N) truyền thống, cung cấp khả năng Rotation Epoch mã hóa cho nhóm lên đến hàng chục nghìn người với chi phí O(log N).
  • Hạn chế: Quá trình Tree Rotation phụ thuộc vào tính toán đồng thời giữa các Nodes. Bắt buộc Server phải vận hành dịch vụ trung gian Delivery Service để gom Commit blob. Các tài khoản Offline lâu ngày cần cơ chế Cross-sign dự phòng từ Desktop ủy quyền chữ ký qua nhánh Mobile.

☁ VPS Cluster · 🗄 Server Vật Lý

  • Cross-platform: Không (Thiết kế Native Linux System)

A. Time Drift & Logical Clocks (Chống sụp đổ E2EE)

Sự lệch trục thời gian giữa thiết bị Mobile và Server VPS rẻ tiền (Time Drift) hoặc việc User cố tình chỉnh sai OS Clock là nguyên nhân chính làm Epoch MLS bị từ chối do Replay Attack verification thất bại. Rust Core tuyệt đối không tin tưởng OS Time. Thay vào đó:

  1. Kéo thời gian chuẩn từ In-house NTP Server riêng mỗi khi socket mở.
  2. Dùng Lamport Vector Clocks (Đồng hồ logic) đếm sự kiện tịnh tiến. Nghĩa là Hệ thống định hình Event A xảy ra trước Event B bằng số đếm (Counter), miễn nhiễm với hiện tượng nhảy cóc thời gian trên máy trạm vật lý.

B. Dynamic CPU Dispatching (Chống SIGILL Crashing)

Cố gắng ép Lõi Rust chạy max-speed ở compiler-level sẽ làm sập VPS cũ hoặc máy tính PC đời Tống. Khi App khởi động (Initiation), hàm FFI sẽ check Hardware Flags (AVX-512? ARM NEON? SSE4.1?). Nếu có, pipeline đi vào "Fast Path". Nếu không, pipeline chuyển mềm sang "Safe Path" (Sofware Encryption) mà không bị hệ điều hành ném lỗi SIGILL.

C. Anycast TURN Cascading (Vượt Symmetric NAT Doanh nghiệp)

Lưới mạng P2P cho TeraVault rất dễ chết lâm sàng ở môi trường công sở dùng NAT cấp doanh nghiệp (Symmetric NAT gắt). Xây ngầm một mạng lưới Global TURN Clusters liên kết chéo trên 1 địa chỉ Anycast duy nhất để làm Relay Proxy kín (Zero-Knowledge) thay vì thất bại khi P2P Hole Punch vô vọng.

D. Object Blob Virtualization (Chống hỏng file do hệ điều hành)

Hệ điều hành Windows có đặc tính Case-Insensitive (không phân biệt hoa thường) và backslash (\), khác máu hoàn toàn với iOS/macOS/Android. Để đồng bộ P2P 1 thư mục chứa hàng chục nghìn file không sụp đổ, TeraChat cấm mọi thao tác ghi tên file thật (như Báo_Cáo_Tài_Chính.pdf) xuống đĩa OS.

Toàn bộ File tải về được "băm" thành d3b07384-d113.bin. Hệ điều hành OS chỉ nhìn thấy "bãi chiến trường" Blob File rác. Việc "vẽ" lại cây thư mục + Emoji Tên file đẹp đẽ được quản lý 100% trong RAM/SQLite (SQLCipher).

  • iOS Background: chỉ đóng vai trò Leaf Node (thụ động) trong BLE Mesh do CoreBluetooth throttle.

💻 Laptop / 🖥️ Desktop (Windows / macOS / Linux):

  • RAM Pinning: VirtualLock() (Windows) / mlock() (macOS, Linux) vùng nhớ Plaintext trong Rust Core.
  • VFS Bridge: SharedArrayBuffer (bypass OPFS) → stream dữ liệu > 500MB/s vào Rust Core.
  • Desktop Daemon: Tiến trình Rust nhẹ (< 5MB RAM) chạy nền, duy trì WebSocket → gửi IPC wake-up lên Tauri UI khi có tin nhắn mới ngay cả khi user tắt UI.
  • Laptop: cài handler lid-close → flush I/O SQLCipher trước khi Sleep/Hibernate.
  • Desktop: Relay Node đầy đủ trong BLE Mesh (cắm điện 24/7, không bị OS throttle).

🗄 Server Vật Lý / ☁ VPS:

  • SMT Solver Formal Verification trong CI/CD pipeline để xác minh logic Protocol MLS không có lỗi logic hình thức.
  • GeoHash Indexing trên OPA giúp tính toán truy cập lân cận với độ trễ < 1ms.
  • Tiêu chuẩn CAP_IPC_LOCK bắt buộc kiểm tra khi boot AI Worker Container.

13.11 Unidirectional State Sync (UI ↔ Rust Core) `📱 Mobile` `💻 Laptop` `🖥️ Desktop`

Khủng hoảng Đồng bộ Trạng thái: Kiến trúc quy định UI (React Native / Tauri) là "Pure Renderer", mọi dữ liệu nằm ở Lõi Rust (SQLCipher). Tuy nhiên, nếu đoạn code UI liên tục lấy mẫu (polling - setInterval) Rust qua FFI để xem có tin nhắn mới không, App sẽ giật lag và hao pin nghiêm trọng. Nếu Rust chủ động "bắn" toàn bộ cục dữ liệu JSON lớn lên UI mỗi khi có event, Main Thread của UI sẽ nghẽn, gây rớt khung hình (khung hình < 60fps), hoặc mất event nếu UI đang bận.

Giải pháp: Unidirectional Data Flow với Versioning

  1. Lightweight Signal (Tín hiệu siêu nhẹ): Khi nhận tin nhắn lưới/WebSocket, Rust Core xử lý xong và lưu vào SQLite. Sau đó, Rust chỉ bắn một event cực nhẹ chứa Metadata qua JSI/IPC: StateChanged(table = "chats", version = 42).
  2. Pull-based Rendering: Component UI bắt được tín hiệu StateChanged. Nó sẽ tự quyết định thời điểm rảnh rỗi (Idle) để chủ động gọi hàm fetch_snapshot("chats") xuyên qua cầu nối SharedArrayBuffer (Desktop) hoặc Pointer (Mobile) để lấy Array dữ liệu mới nhất.
  3. Lợi ích: UI Thread không bao giờ bị nghẽn bởi payload rác. Rust không cần quan tâm UI đang làm gì. Không bao giờ mất đồng bộ trạng thái.

13.12 KMS Bootstrap & Onboarding Security (Admin Kém Tắm) `⚙️ Core` `☁️ Server`

Thảm họa "Zero-Access": Quản trị viên IT (Admin) thiết lập hệ thống thường không phải là chuyên gia mật mã. Việc tự cấu hình Master Key, Recovery Key hay Federation JWT nếu phức tạp sẽ dẫn tới rủi ro: Admin làm mất Master Key, và hệ thống Zero-Knowledge mã hóa toàn bộ cơ sở dữ liệu vĩnh viễn, tước đi quyền truy cập của chính công ty đó.

Giải pháp: Ép buộc bằng UX Constraint (KMS Bootstrap)

  • Trong màn hình thiết lập Init_Workspace đầu tiên, App sẽ sinh ra một tập tin vật lý tên là terachat_master_<domain>.terakey (bản chất là Master Key bọc AES-256 nội bộ với mật khẩu mạnh của Admin).
  • Lõi Rust khước từ (Block) tiến trình tạo Database cho đến khi Admin thực sự nhấn chọn thư mục để "Lưu trữ File Key Backup" này hoặc "In ra giấy" (Print as QR/Text).
  • Backup File này là "Phao cứu sinh" duy nhất nếu VPS bị cháy ổ cứng hoặc Admin mất mật khẩu đăng nhập. Việc chuyển giao trách nhiệm lưu trữ vật lý bằng một tệp .terakey cụ thể sẽ giúp "Admin Kém Tắm" cũng hiểu được mức độ quan trọng sống còn của chìa khóa mã hóa.

13.13 Final Enterprise Recommendation `⚙️ Core`

🔺 IMPORTANT

Nguyên tắc bất khả xâm phạm của kiến trúc TeraChat Alpha:

1. Ngừng mọi nỗ lực port logic Crypto/Business Logic lên JS/Dart Thread. Việc duy nhất của UI (React Native / Flutter / Tauri WebView) là render. Không hơn.

2. Áp dụng triệt để Control Plane / Data Plane Separation cho mọi giao tiếp IPC:

- Control Plane: Protobuf message qua JSI HostObject / Dart FFI (lệnh điều khiển nhỏ, < 1KB).

- Data Plane: SharedArrayBuffer (Desktop) / C++ Shared Memory (Mobile) — Zero-copy cho payload lớn.

3. Rust Core nắm giữ toàn bộ sinh mệnh hệ thống: MLS, Keystore, Storage I/O, Network, OPA Policy. Không có exception.

4. Server là Blind Relay. Server không được phép biết nội dung của bất kỳ message nào. Kiến trúc Zero-Knowledge không phải tham vọng, mà là giới hạn thiết kế cứng.

5. Hardware HSM / Secure Enclave cho mọi thiết bị. Private Key không bao giờ rời chip. ID của Key được truyền thay vì raw pointer. Đây là ranh giới không thể thương lượng.

Tài liệu này tổng hợp và mở rộng các quyết định kiến trúc từ §1–§12. Mọi thay đổi kiến trúc ảnh hưởng đến nguyên tắc Zero-Knowledge hoặc Key Management phải được Chief Enterprise Architect phê duyệt trước khi triển khai.

14. ENTERPRISE RELIABILITY & LIFECYCLE MANAGEMENT

Kiến trúc quản trị vòng đời ứng dụng, giới hạn tài nguyên, và kịch bản thảm họa (Disaster Recovery) nhằm đảm bảo hệ thống không thể bị sập, đánh cắp dữ liệu bằng mẹo (trick), hay "đột tử" khi scale lên quy mô hàng triệu users.

14.1 OTA & Downgrade Prevention (Version Handshake) `📱 Mobile` `💻 Laptop` `🖥️ Desktop` `☁️ Server`

Thiệt hại nếu thiếu: Kẻ tấn công cố tình cài phiên bản TeraChat Client cũ đã bị công bố lỗi nắn dòng (buffer overflow) để bypass các rules OPA Policy trên thiết bị mới.

Giải pháp: Bức tường Checksum Version Validation

  1. Handshake Bắt buộc: Bất cứ khi nào Client kết nối WebSocket lên Server, nó phải gửi kèm App_Version ký bằng JWT. Server sẽ đối chiếu với Min_Required_Client_Version lưu trên Database.
  2. Hard-Lock UI: Nếu Client Version < Min_Required, Rust Core sẽ tự động ném Error::ForceUpdateRequired, khóa cứng React Native/Tauri UI (chỉ hiện màn hình Yêu cầu Cập nhật). Quan trọng hơn, Rust sẽ từ chối giải mã SQLite Database nội bộ, triệt tiêu mọi khả năng đọc trộm dữ liệu offline từ bản cũ.

14.2 WASM Sandbox Resource Watchdog `📱 Mobile` `💻 Laptop` `🖥️ Desktop`

Thiệt hại nếu thiếu: Một tiện ích nội bộ (CRM / HR .tapp) bị code dính vòng lặp vô hạn while(true) hoặc cấp phát mảng khổng lồ làm vắt kiệt RAM, chết Main Thread, sập pin điện thoại.

Giải pháp: Execution Quota Throttling

  1. CPU Watchdog: Wasmtime/WASM Runtime trong Lõi Rust được cấu hình Timer ngắt mạch. Thời gian thực thi (Execution time) cho mọi hàm FFI .tapp gọi xuống Rust bị giới hạn tối đa 50ms. Vượt quá -> Kill Thread (TAPP_CPU_TIMEOUT).
  2. Memory Quota: Cấp phát vRAM tối đa 100MB cho mỗi Wasm Module Instance. Cố gắng allocate quá mức -> Kill Thread (TAPP_OOM). Tiện ích bên thứ 3 chết, phần chat cốt lõi của TeraChat vẫn mượt mà.

14.3 Non-GMS Push Notifications (Vùng trắng Push) `📱 Mobile` `🚀 VPS`

Thiệt hại nếu thiếu: Điện thoại Android nội địa Trung Quốc, Huawei (HMS) không có Google Play Services. Do đó FCM (Firebase Cloud Messaging) bị cụt, user bị trễ tin nhắn.

Giải pháp: Long-polling TCP / MQTT Keep-alive

  1. Dedicated MQTT Channel: Cho các máy Non-GMS, Rust Core sẽ duy trì một luồng TCP socket tĩnh nhẹ (ping payload 1 byte/phút).
  2. Wake-lock Throttling: Kết hợp Doze Mode của Android bằng cách nhóm (batching) các bản ping theo chu kỳ 5 phút/lần khi màn hình tắt để cứu pin, chỉ giữ kết nối thời gian thực khi user mở sáng màn hình.

14.4 Edge-Side Media Compression `📱 Mobile` `💻 Laptop` `🖥️ Desktop`

Thiệt hại nếu thiếu: Hàng ngàn nhân viên ném thẳng video 4K 2GB hoặc ảnh RAW 50MB hệ thống E2EE. Server MinIO phình to vô ích, Cloud Egress Bandwidth Cost tăng kỷ lục, mạng P2P nghẹt thở.

Giải pháp: Nén bắt buộc trước mã hóa (Pre-encryption Compression)

  1. Image Engine: Tích hợp bộ nén vào WASM hoặc Native OS Codec. Cưỡng ép mọi ảnh (trừ file Document PDF/Word) phải được resize và nén sang định dạng WebP hoặc HEIC ở mức quality 80%.
  2. Video Engine: Bọc FFmpeg (tối giản) qua luồng C++ / Rust. Resize phần cứng và Transcode mọi video về chuẩn H.265 (HEVC) hoặc tối đa H.264 720p trước khi dòng Byte đi vào hàm aes_256_gcm_encrypt. Không có ngoại lệ. File đã bị ném vào Rust là file đã nhỏ gọn.

14.5 PostgreSQL Disaster Recovery (PITR) `☁️ Server` `🚀 VPS`

Thiệt hại nếu thiếu: HA (High Availability) chỉ chống được phần cứng sập. Nếu admin lỡ tay gõ lệnh DELETE FROM users; hay DROP TABLE, Postgres HA sẽ lập tức đồng bộ lệnh xóa tàn phá đó sang tất cả các Node Standby. Không thể cứu vãn Metadata.

Giải pháp: Cơ chế WAL Archiving (Point-in-Time Recovery)

  1. Mọi lệnh UPDATE/DELETE trên Primary Node không chỉ ghi vào bảng, mà còn nhả ra các tệp WAL (Write-Ahead Logging).
  2. Daemon pgBackRest hoặc WAL-G trên Server sẽ tự động bưng dọn các file WAL này (mỗi chu kỳ 5 phút/lần) lên một Hidden MinIO Storage Bucket hoàn toàn độc lập với Cluster hiện tại (Read-Only Mode).
  3. Khi xảy ra thảm họa thao tác nhầm, Admin chỉ cần chỉ định "Hồi sinh database vào lúc 14:03:00". Script sẽ dựng một Postgres Server mới, nạp bản Full Backup rạng sáng, rót tuần tự các file WAL vào cho đến đúng cột mốc 14:03:00 thì ngắt. Toàn vẹn 100% dữ liệu.
